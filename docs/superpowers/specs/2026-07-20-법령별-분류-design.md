# T20 설계 — 문제 분류를 법령(과목) 중심으로 변경

- 작성일: 2026-07-20
- 태스크: T20 (`소방-기출앱`)
- 상태: 설계 확정(사용자 승인 2026-07-20)

## 1. 배경 / 목표

문제 탐색의 1차 분류를 **법령별**로 재구성한다. 태스크 제목은 "연도별 → 과목별"이지만, 탐색 결과 **현재 분류는 실제로 연도(`year`) 기반이 아니다.** 실제 1차 분류는 런타임에 문항 태그로 생성되는 `QuestionCollection`(회차 `2026-N회`=원형 / `심화` / `전체`)이며, `year` 정수 필드는 그룹핑·조회·이어풀기 어디에도 쓰이지 않는다("연도별"은 회차 태그에 연도가 들어 있어 생긴 착시).

또한 "과목(subject)" 계층은 이미 코드에 존재하나(`Subject`, `subjectId`, `getSubjects()`) 콘텐츠가 소방관계법규(`fire-law`) 1개뿐이라 화면에서 자동 펼쳐진다.

**이 태스크에서 "과목" = 소방관계법규를 이루는 개별 법령.** 문항을 법령 카테고리로 묶어 1차 브라우즈 축으로 삼는다.

## 2. 확정 결정 (브레인스토밍 2026-07-20)

1. **"과목" = 6개 법령**: 소방기본법 · 화재예방법 · 소방시설법 · 소방공사업법 · 위험물안전관리법 · 화재조사법.
2. **카테고리 8개**: 위 6개 법령 + `교차법령` + `심화 OX·계산`(법령 미분류 심화 문항). 6개 법령에 안 들어가는 57문항(교차 39 + 미분류 18)을 각각 별도 카테고리로 둔다(콘텐츠 손실 없음).
3. **법령 하위 구조 없음**: 법령 선택 → 그 법령의 원형+심화 문항을 병합해 바로 풀이. 원형/심화·회차는 태그로만 보존(탐색 동선에서 제거).
4. **구현 접근 = 카테고리 개념 정식 도입 + 명명 정리**: 실제로 collection id를 담고 있던 `subjectId` 명명 혼선을 `categoryId`로 정리.
5. **콘텐츠 불변**: `assets/content/fire-law.json` 미수정. 법령 분류는 기존 태그로 파생하고, 소방시설법 이름 변형은 코드에서 정규화.

## 3. 개념 모델 & 명명 정리

- **Subject**(소방관계법규, id `fire-law`) — 그대로 유지. 콘텐츠 최상위(현 1개, 휴면 계층).
- **`QuestionCollection` → `QuestionCategory` 리네임** (`{id, name, group, count}`). 1차 브라우즈 단위 = 법령 카테고리.
- **`subjectId`(실제로는 category id를 담던 곳) → `categoryId` 리네임 대상**:
  - `GetQuestionSet.call(categoryId, mode)`
  - `QuizArgs.categoryId`
  - `Routes.quizLink/examLink` + 쿼리 파라미터, `app_router.dart`
  - `home_view_model`의 `resumeInfoProvider`·`subjectStatsProvider` 등
  - `Sessions.key`·`RecentSession` 매핑
- **⚠️ 유지(리네임 금지) — 진짜 subject(`fire-law`)를 담은 곳**: AI 문제함/생성 경로의 subjectId — `aiBankCountProvider`(하드코딩 `'fire-law'`), `GeneratedQuestions.subjectId`, `PendingAiRequests.subjectId`. 이들은 실제 과목이라 카테고리와 무관. **구현 시 각 `subjectId` 사용처가 category를 뜻하는지 subject를 뜻하는지 개별 판별하는 것이 최우선 주의점.**

## 4. 카테고리 정의 (코드 상수)

코드에 고정 목록으로 정의(런타임 태그 수집이 아니라 명시적 상수 — 법령 집합이 고정이므로).

- 표시 순서: 소방기본법 → 화재예방법 → 소방시설법 → 소방공사업법 → 위험물안전관리법 → 화재조사법 → 교차법령 → 심화 OX·계산.
- `group`: 6개 법령 = `"법령"`, 교차법령·심화 OX·계산 = `"기타"`.
- id 스킴 예: `fire-law::law:기본`, `fire-law::law:시설`, `fire-law::cross`, `fire-law::simhwa-etc`. (기존 `subject::filter` 스킴과 호환되는 형태.)

**분류 함수 `classify(question) -> categoryId`** — 문항당 정확히 1개:
1. `교차법령` 태그 있음 → 교차법령
2. 아니면 법령 태그(정규화 후) 있음 → 그 법령
3. 둘 다 없음 → 심화 OX·계산

**소방시설법 정규화**: 변형 5종을 코드 맵으로 하나의 소방시설법 카테고리에 흡수(JSON 미수정).

**"전체" 카테고리 유지**: id `fire-law`, name `전체`. 법령 목록과 별도로 취급하며 홈의 빠른/랜덤/오답/진척 분모 및 전체풀기에 사용.

## 5. 레이어별 변경

### 데이터/도메인
- `QuestionRepository.getCollections()` → `getCategories()`; `getQuestions(categoryId)` 매칭을 `classify` 기반으로 재작성(현행 `subject::filter` 태그 매칭 대체).
- `QuestionRepositoryImpl`: 회차 수집(`_roundTag`)·심화(`_srcTag`) 로직을 카테고리 상수 + `classify`로 교체.
- `ContentDataSource`: 단일 파일 로드 유지(다과목 확장 구조 그대로). `year`는 메타로만 보존.

### 프레젠테이션
- `subjects_page`: 8개 카테고리를 2섹션(`법령` 6 / `기타` 2)으로 렌더. 기존 `group` 기반 섹션 분리 재사용(group 값만 재정의), 태블릿 2열 그리드 유지.
- `home_page`/`home_view_model`: "전체" 선택 로직을 categoryId로. 하드코딩 `'fire-law'`가 true subject인 곳은 유지.
- `set_mode_page`·`quiz_view_model`·`app_router`: 파라미터명 categoryId로 리네임(동작 동일).

### 저장·이어풀기 마이그레이션
- 세션키 규칙 `"$categoryId:normal"`. 기존 회차기반 세션키(`fire-law::2026-1회:normal` 등)는 새 카테고리와 안 맞아 무효.
- **drift `schemaVersion` 3→4, 마이그레이션에서 `Sessions` 테이블 클리어**(개발 단계 앱, 진행중 이어풀기 소실 감수).
- 진척(`AttemptRecords`)·오답(`WrongEntries`)은 **전역 유지**(현행과 동일, 스코프 변경 없음).

## 6. 에러 처리 & 엣지 케이스

- 알 수 없는 categoryId → 빈 문제 세트 반환(크래시 없이 빈 상태 UI).
- 빈 카테고리: 현 데이터상 전 카테고리 ≥7문항이라 없음. 그래도 count 0 카테고리는 목록에서 숨김(방어).
- 태그 변형: 소방시설법 5종 정규화. 그 외 예기치 못한 법령 태그가 생기면 `classify`의 3단계(심화 OX·계산)로 안전 낙하.

## 7. 테스트 (완료기준)

- `classify` 단위 테스트: 각 법령 태그 → 올바른 카테고리, 교차법령 → 교차, 법령 없는 심화 → 심화 OX·계산, 소방시설법 변형 5종 → 소방시설법. **카운트 46/49/71/50/53/7/39/18 = 333 검증.**
- `getCategories`: 8개 반환, `group`·순서·count 정확.
- `getQuestions(categoryId)`: 올바른 pool. **AC 핵심 — 소방기본법 카테고리에 2026 회차(원형)와 2025(심화) 문항이 함께 포함**됨을 검증(여러 연도 병합).
- 일반·랜덤·빠른·시험·이어풀기가 categoryId 기준으로 동작.
- `flutter analyze` 이슈 0 + 전체 테스트 통과.

## 8. 범위 밖 (non-goal)

- `year` 필드 삭제, 연도별 필터 신규 제작.
- 다과목(소방학개론 등) 콘텐츠 추가 — subject 계층은 그대로 두되 이번엔 소방관계법규 1개.
- 콘텐츠(JSON) 지문·정답·해설·태그 수정.
- 법령별 진척/오답 스코핑(전역 유지).
- 원형/심화·회차 하위 필터 UI.

## 9. 리스크 / 구현 주의점

- **`subjectId` 이중 의미**: category vs true-subject 판별을 사용처별로 정확히 — 오판 시 AI 문제함·생성이 깨지거나 세션키가 뒤섞임. §3 유지 목록 준수.
- **drift 스키마 마이그레이션**: schemaVersion 상향 + `build_runner` 재생성 필요. 마이그레이션 로직 검증(기존 설치에서 Sessions만 클리어, 나머지 보존).
- 명명 리네임 diff가 넓음 — 회귀 방지 위해 각 단계 후 `flutter analyze`·테스트로 게이트.
